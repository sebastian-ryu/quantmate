# 프로젝트 개요

## 목표

주식 투자 연구, 종목 선정, 전략 백테스트, 향후 한국투자증권 연동 기반 매매를 지원하는 로컬 웹 애플리케이션을 만든다.

첫 번째로 쓸 수 있는 제품은 설명 가능한 전략으로 종목을 찾고, 그 전략을 백테스트로 검증할 수 있어야 한다. 한국투자증권 모의투자와 실거래 자동 매매는 이후 단계이며 강한 안전장치가 필요하다.

## 제품 우선순위

1. 전략 메뉴를 중심으로 시스템 제공 전략과 사용자 등록 전략을 관리한다.
2. 실제 트레이더 전략과 유명 퀀트 전략을 초기 시스템 제공 전략으로 만든다.
3. 선택한 전략으로 종목 후보를 검색하고, 왜 후보가 되었는지 설명한다.
4. 선택한 전략을 백테스트로 검증한다.
5. 전략과 백테스트가 충분히 동작한 뒤 상세 검색기를 실제 데이터에 연결한다.
6. 주식 관련 데이터를 수집하고 정규화한다.
7. 재사용할 데이터는 가능한 한 로컬에 저장한다.
8. 한국투자증권 Open API로 초기 실제 데이터, 실시간 데이터, 향후 매매 기능을 연동한다.
9. KRX Open API는 승인 후 백테스트용 공식 장기 데이터 공급원으로 붙이고, 승인 지연 시 Yahoo/yfinance를 임시 백테스트 데이터로 사용한다.
10. 한국투자증권 모의투자 계좌 연동을 실거래 전 검증 단계로 둔다.
11. 한국 주식을 먼저 지원한다.
12. 미국 주식 확장은 염두에 두되 초기에는 만들지 않는다.

## 첫 버전에서 하지 않을 일

- 모바일 앱 지원
- 클라우드 배포
- 다중 사용자 계정 관리
- 처음부터 완전 자동 실거래
- 첫 버전의 완전한 웹 기반 전략 빌더
- 기본 종목 선정과 백테스트 이전의 고급 포트폴리오 최적화
- 무료/공개 데이터로 부족하다는 판단 전의 유료 데이터 벤더 연동

## 주요 위험

- 실거래 자동화로 인한 금전 손실
- 데이터 제공처별 데이터 품질 차이
- 상장폐지 종목과 액면분할 등 기업 이벤트를 무시한 백테스트 왜곡
- API 호출 제한과 증권사 정책 변경
- API 키 등 비밀 정보 유출
- 과거 데이터에만 과도하게 맞춘 전략

## 안전 기준

실거래는 마지막 단계로 둔다.

초기 시스템은 다음 모드를 지원하는 구조로 설계한다.

1. `research`: 데이터 수집, 종목 선정, 백테스트만 수행
2. `kis-readonly`: 한국투자증권 계좌와 시장 데이터를 읽기만 함
3. `kis-paper-trading`: 한국투자증권 모의투자 계좌에 주문 제출
4. `kis-live-trading`: 실제 주문, 기본 비활성화, 강한 제한 필요

조건 검색, 퀀트 전략, 자동매매 전략은 별도 개념으로 흩뜨리지 않고 하나의 전략 정의에서 출발한다. 같은 전략 정의를 `research`, `backtest`, `kis-paper-trading`, `kis-live-trading` 실행 모드가 공유하되, 실제 주문을 내는 실행 계층은 주문 한도, 중복 주문 방지, 체결 확인, 손실 제한, 긴급 중지 같은 안전장치를 별도로 가진다.

## 초기 아키텍처 방향

처음에는 모듈형 단일 애플리케이션으로 시작한다.

- 웹 클라이언트: SvelteKit
- 백엔드 API와 작업 조율: Python + FastAPI
- 데이터/전략/백테스트 실행: 같은 코드베이스 안의 Python 모듈
- 데이터베이스: 앱 데이터와 정규화된 시장 데이터를 저장하는 로컬 MySQL
- 증권사 연동: 한국투자증권 Open API

MVP에서는 백엔드, 데이터 수집, 전략 로직, 백테스트를 Python 하나로 묶어 복잡도를 줄인다. Java/Spring Boot는 서버 규모가 커져 더 강한 엔터프라이즈 구조가 필요할 때 다시 검토한다.
